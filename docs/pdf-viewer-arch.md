# PDF Viewer Architecture

This document describes the current PDF viewer rendering flow in `md-editor`.
It focuses on the parts that matter for navigation reliability: worker scheduling,
page rendering, pending state, caching, link extraction, and scroll handling.

## Overview

The PDF viewer is split across two crates:

1. `md-editor-core`: owns the PDFium integration in `core/src/pdf.rs`.
2. `native`: owns iced UI state, messages, rendering tasks, and scroll behavior in
   `native/src/app.rs`.

PDFium is treated as a single-worker resource. The application must not create
multiple `Pdfium::new()` instances because `pdfium-render` stores the bindings in
a process-global cell. The renderer therefore uses one background worker thread
with an internal priority scheduling loop.

```text
iced UI
  -> Message handlers in native/src/app.rs
  -> iced::Task async calls
  -> PdfRenderer API in core/src/pdf.rs
  -> single PDFium worker
  -> typed Message result back to app.rs
  -> generation validation, pending cleanup, cache update, optional follow-up task
```

## Core Renderer

`PdfRenderer` lives in `core/src/pdf.rs`.

It exposes synchronous request methods used from iced tasks:

- `page_count(path)`
- `page_sizes(path)`
- `get_toc(path)`
- `render_page(path, page, scale)`
- `render_page_priority(path, page, scale)`
- `get_page_links(path, page)`
- `search_text(path, query, regex, match_case)`
- `render_link_preview(path, page, dest_y)`

Internally it owns:

- `sender: Sender<PdfCommand>` for normal work.
- `priority_sender: Sender<PriorityRender>` for target page renders.
- one worker thread with one PDFium binding and one cached `PdfDocument`.

The worker keeps:

```rust
current_document: Option<(String, PdfDocument)>
pending_commands: VecDeque<PdfCommand>
```

The scheduling rule is:

1. Receive normal commands from the channel.
2. Drain currently available normal commands into `pending_commands`.
3. Before each normal command is handled, drain priority requests.
4. If multiple priority requests are waiting, render only the newest one.
5. Process one normal command.
6. Repeat.

`PdfCommand::Wake` exists only to unblock `receiver.recv()` after a priority
request is sent. The `Wake` command itself performs no work.

This design cannot interrupt an already-running PDFium operation. It does ensure
that a TOC target page does not wait behind already queued preloads, link
extraction, search, or previews once the current operation completes.

## Native PDF State

`MdEditor` tracks PDF UI state in `native/src/app.rs`.

Important fields:

- `pdf_current_page`: current page according to scroll/navigation state.
- `pdf_total_pages`: page count from `PdfLoaded`.
- `pdf_zoom`: active render scale.
- `pdf_pages`: cached rendered page image handles.
- `pdf_dimensions`: cached rendered image dimensions.
- `pdf_page_sizes`: unrendered PDF page sizes from PDFium, in PDF points.
- `pdf_placeholder_page_size`: one stable unscaled page size used by all
  unloaded blank placeholders.
- `pdf_scroll_y`: current scroll offset.
- `pdf_page_links`: loaded links per page.
- `pdf_page_text`: lazily loaded text geometry for visible pages.
- `pdf_annotations`: sidecar study highlights and notes grouped by page.
- `pdf_selection`: active PDF text selection in page text-index coordinates.
- `pdf_pending_pages`: pages with image render requests in flight.
- `pdf_pending_links`: pages with link extraction requests in flight.
- `pdf_pending_text`: pages with text extraction requests in flight.
- `pdf_render_generation`: invalidates stale async results after document,
  zoom, or navigation changes.
- `pdf_programmatic_scroll`: suppresses normal viewport scheduling for the
  scroll event generated by a TOC/link/search jump.
- `pdf_toc_target_page`: target page currently being navigated to.

## Generation And Pending Invariants

Render generation is used to ignore stale results. It must not be used to skip
cleanup.

The invariant is:

```rust
Message::PdfRendered(generation, page, img) => {
    self.pdf_pending_pages.remove(&page);

    if generation != self.pdf_render_generation {
        return Task::none();
    }

    // update page cache
}

Message::PdfRenderFailed(generation, page) => {
    self.pdf_pending_pages.remove(&page);

    if generation != self.pdf_render_generation {
        return Task::none();
    }

    // report active-generation failure
}
```

Similarly, link completion removes `pdf_pending_links` before checking
generation:

```rust
Message::PdfPageLinksLoaded(generation, page, links) => {
    self.pdf_pending_links.remove(&page);

    if generation != self.pdf_render_generation {
        return Task::none();
    }

    self.pdf_page_links.insert(page, links);
}
```

Any code that increments `pdf_render_generation` should also clear in-flight
state that belongs to the previous generation:

- `pdf_pending_pages.clear()`
- `pdf_pending_links.clear()`

Document changes also clear `pdf_page_links`, page images, rendered dimensions,
and page-size metadata.

## Page Rendering

Normal page rendering is scheduled through:

- `render_visible_pdf_pages()`
- `render_pdf_pages_for_viewport(scroll_y, viewport_height)`
- `render_pdf_page_range(start, end)`
- `render_pdf_page(page)`

`render_pdf_page_range()` deduplicates image jobs using `pdf_pages` and
`pdf_pending_pages`:

```rust
if page image is missing && page is not pending {
    pdf_pending_pages.insert(page);
    render_pdf_page(page)
}
```

`render_pdf_page()` renders only the page image. The requested PDFium render
scale is supersampled relative to the displayed zoom so narrow split-view pages
remain sharp when drawn into the page bounds. The cached `pdf_dimensions` remain
logical display dimensions, not raw bitmap dimensions, so link hit-testing and
placeholder fallback math continue to use PDF-space/display-space geometry.

`render_pdf_page()` does not extract links.
If the renderer is missing or rendering fails, it returns `PdfRenderFailed`
instead of hiding the issue behind `Message::Tick`.

When `PdfRendered` succeeds:

1. Remove the page from `pdf_pending_pages`.
2. Ignore stale generations.
3. Store the image handle in `pdf_pages`.
4. Store `(width, height)` in `pdf_dimensions`.
5. Schedule low-priority link extraction and text extraction for that page.
6. If the page is the active navigation target, compute the scroll offset and
   issue the programmatic scroll.

## Text Selection And Study Overlays

PDF text extraction is loaded lazily through `get_page_text(path, page)`.
Extraction runs on the same single PDFium worker as rendering, so it is
deduplicated with `pdf_pending_text` and bounded by the visible page range.
Text is stored in PDF-space coordinates and survives zoom changes.

`native/src/views/interactive_pdf.rs` draws the page bitmap plus overlay layers
inside one custom widget:

1. Persistent sidecar annotation highlights.
2. Active PDF search highlights.
3. Active text-selection highlight.

This avoids creating one iced widget per annotation rectangle and keeps large
study documents responsive.

PDF highlights, quick notes, and linked markdown notes are stored in SQLite
sidecar tables. Original PDFs are not modified. Linked markdown notes use the
normal vault files and can participate in the existing markdown/backlink graph.

## Page Size Metadata

The viewer loads every page's unscaled PDF size through `page_sizes(path)` when
a PDF opens. These sizes are much cheaper than rendering page images and are
available before most pages enter the render queue.

Page-size metadata is used to establish a stable placeholder size, normally from
the first page. Unloaded pages are laid out as blank pages with that same
placeholder size.

- loading placeholder dimensions;
- `pdf_page_offset(page)`;
- `pdf_page_at_scroll(scroll_y)`;
- viewport range detection;
- fit-to-width zoom calculation.

Every page occupies the same outer slot while scrolling, including pages whose
bitmaps have already rendered. Rendered bitmap dimensions are cached for image
data and link calculations, but they do not change the scroll geometry. The page
slot size is computed as:

```text
(pdf_placeholder_width * pdf_zoom, pdf_placeholder_height * pdf_zoom)
```

This keeps the scroll model aligned with the actual placeholder layout. Without
stable placeholder geometry, jumps to a page deep in a PDF can accumulate offset
errors because the scroll model and the actual blank-page layout can diverge.

The fixed-slot behavior is covered by unit tests in `native/src/app.rs`. These
tests assert that target offsets map back to the same page, blank pages reserve
space in total document height, and placeholder slots scale with zoom.

## Link Extraction

Links are loaded separately from page images through `load_pdf_page_links(page)`.

This prevents link extraction from keeping a page in a visual loading state.
The page image and page links use separate pending sets:

- `pdf_pending_pages` for rendered images.
- `pdf_pending_links` for link metadata.

`load_pdf_page_links()` deduplicates requests using `pdf_page_links` and
`pdf_pending_links`.

Links are generally loaded after the page image renders. This keeps navigation
responsive and avoids scheduling image rendering and link extraction for many
nearby pages at the same time during TOC jumps.

## TOC, Link, And Search Navigation

PDF navigation is centralized through `navigate_pdf_page(page)`.

It is used by:

- TOC clicks.
- Internal PDF link clicks.
- PDF search result navigation.

Navigation flow:

1. Clamp and store `pdf_current_page`.
2. Clear `pdf_pending_pages` and `pdf_pending_links`.
3. Increment `pdf_render_generation`.
4. Store `pdf_toc_target_page`.
5. Schedule link loading if the target page is cached, otherwise schedule the
   target page through `render_pdf_page_direct()`.
6. Compute the target offset using the same stable blank-page slot dimensions
   used by the view.
7. Set `pdf_programmatic_scroll = true` and issue `scroll_to`.

`render_pdf_page_direct()` calls `render_page_priority()` and marks the target
page pending if its image is missing.

When the target page render completes, `PdfRendered` stores the bitmap and
dimensions, but it does not change the page's outer scroll slot.

## Programmatic Scroll Handling

`PdfScrolled { y, viewport_height }` is emitted by iced when the PDF scrollable
moves.

For normal user scrolling:

1. Update `pdf_scroll_y`.
2. Estimate the current page from `y + viewport_height * 0.33`.
3. If the page jump is large, clear pending image/link work.
4. Schedule viewport pages through `render_pdf_pages_for_viewport()`.

For TOC/link/search navigation:

1. `pdf_programmatic_scroll` is set before `scroll_to`.
2. The first matching `PdfScrolled` acknowledges that programmatic scroll.
3. Normal viewport scheduling is suppressed for that event.
4. Only a small nearby range around the target page is scheduled.
5. `pdf_toc_target_page` is cleared.

This prevents a jump from scheduling the target page, nearby pages, viewport
pages, and link extraction all at once.

## Viewport Rendering

`render_pdf_pages_for_viewport(scroll_y, viewport_height)` computes a window
around the visible area:

```text
first = page_at(scroll_y - estimated_page_height)
last = page_at(scroll_y + viewport_height + estimated_page_height)
range = first - 2 .. last + 2
```

The range is intentionally bounded and deduplicated by `render_pdf_page_range()`.
Normal scrolling should preload nearby pages smoothly without rendering the
entire document. Its page lookup uses per-page metadata when available instead
of assuming all pages are letter-sized.

## Zoom And Document Changes

Zoom changes and fit-to-width changes invalidate rendered page images because
the bitmap scale changes.

On zoom changes:

- set the new zoom;
- reset `pdf_pages`;
- reset `pdf_dimensions`;
- clear `pdf_pending_pages`;
- clear `pdf_pending_links`;
- clear programmatic navigation state;
- increment `pdf_render_generation`;
- render visible pages.

Opening a new PDF additionally clears:

- page image vectors;
- dimension vectors;
- page links;
- current page;
- TOC/navigation state.

## UI Components

`native/src/views/pdf_viewer.rs` builds the PDF UI:

- `toolbar`: page indicator, zoom controls, fit control, TOC toggle.
- `search_bar`: PDF search controls and match count.
- `view_continuous`: vertical list of page images/loading placeholders and
  search highlight overlays.

`native/src/views/interactive_pdf.rs` wraps rendered pages in an interactive
iced widget. It draws page bitmaps, overlays search highlights, and emits:

- `PdfLeftClicked(page, x, y)`
- `PdfRightClicked(page, x, y)`

The click coordinates are normalized relative to the displayed page and later
mapped to PDF-space coordinates using cached page dimensions and zoom.

## Reliability Rules

When changing this code, preserve these rules:

1. Do not create more than one `Pdfium::new()` binding in the process.
2. Target page renders must use `render_page_priority()`.
3. Priority rendering may use `Wake`, but `Wake` should not do real work.
4. Remove pending page/link state before generation checks.
5. Stale results must not update UI caches.
6. Link extraction must not be coupled to page image pending state.
7. TOC/link/search navigation should schedule the target first, then scroll,
   then schedule nearby pages.
8. Programmatic scroll events should not trigger the full viewport scheduling
   path.
9. Rapid navigation should behave as latest request wins where practical.
10. Avoid fixes that render every page, sleep, clear all cached pages on every
    scroll, or ignore generation checks.
